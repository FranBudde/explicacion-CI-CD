CI/CD con GitHub Actions, para el stack que uses

Un recorrido completo: qué es CI/CD, cómo funciona GitHub Actions por dentro, y cómo armar un pipeline real que compila tu proyecto, genera el artefacto desplegable y lo lleva por SSH al servidor correcto según el ambiente — con aprobación manual antes de tocar producción. Los conceptos son universales; elegí tu lenguaje arriba y los ejemplos de código se adaptan.

00 · Antes de empezar

Cómo usar este curso

Está pensado para leerse en orden: cada sección se apoya en la anterior. Los bloques de código YAML son reales y funcionan tal cual (los podés copiar con el botón Copiar); los comentarios dentro del código explican cada línea. Al final hay un workflow completo que combina todo: push → compilar → testear → generar el artefacto → desplegar por SSH al servidor del ambiente que corresponda.

El selector de arriba ("Ver ejemplos para:") controla los ejemplos específicos de cada tecnología: comandos de build y tests, empaquetado, script de despliegue y el workflow final. Todo lo conceptual —triggers, runners, environments, secrets, rollback, seguridad— es idéntico para cualquier lenguaje; por eso el 80% del curso no cambia al cambiar de stack. Si usás otro lenguaje (PHP, Ruby, Rust…), el patrón es exactamente el mismo: cambiá la action de setup-* y los comandos de build, y el resto queda igual.

¿Tu equipo usa GitLab en vez de GitHub? Leé el curso igual: los conceptos son los mismos, y la sección 18 traduce todo a GitLab CI/CD, con el pipeline equivalente completo.

Y cuando termines de leer, no te quedes ahí: la sección 19 es una guía de 20 ejercicios, de básicos a avanzados, para practicar cada tema con las manos. Leer este curso te hace entender CI/CD; los ejercicios te hacen saber hacerlo.

Las versiones de las actions (@v4, @v5, @v1) son las líneas estables y documentadas a mediados de 2026. Estas versiones "mayores" se actualizan seguido (por ejemplo, actions/checkout ya publicó v7 y upload-artifact v7): en la sección 16 te muestro cómo mantenerlas al día automáticamente con Dependabot, en lugar de perseguirlas a mano.

01 · Fundamentos

Qué es CI/CD

Hoy tu flujo probablemente sea: programás, compilás o preparás la app en tu máquina, generás el paquete desplegable (un .war, un bundle de Node, un binario…), lo copiás al servidor a mano, reiniciás el servicio y cruzás los dedos. CI/CD es automatizar ese camino completo para que cada cambio de código recorra siempre los mismos pasos, verificados, sin intervención humana (o con la mínima que vos decidas).

La sigla junta tres prácticas distintas que conviene separar:

  • Integración Continua (CI): cada vez que alguien sube código, una máquina lo compila y corre los tests automáticamente. El objetivo es detectar roturas a los minutos del commit, no dos semanas después cuando alguien intenta armar el release. "Integrar" viene de integrar el trabajo de todos en una rama común, seguido.
  • Entrega Continua (Continuous Delivery): además de compilar y testear, el pipeline deja listo el artefacto desplegable en cada pasada. Desplegar a producción sigue siendo una decisión humana (un botón, una aprobación), pero es trivial porque todo lo anterior ya está automatizado y probado.
  • Despliegue Continuo (Continuous Deployment): se elimina también ese botón. Si el código pasa todos los controles, va a producción solo. Requiere mucha confianza en tus tests; casi nadie arranca por acá.
Para tu caso

El plan de este curso es: despliegue continuo a dev (cada push despliega solo), y entrega continua a staging y producción (el pipeline prepara todo, pero producción espera tu aprobación manual). Es el balance más usado en la industria.

02 · La herramienta

Anatomía de un workflow de GitHub Actions

GitHub Actions es el sistema de CI/CD integrado en GitHub. No tenés que instalar ningún servidor: escribís un archivo de texto en tu repo y GitHub se encarga de ejecutarlo. El vocabulario, de afuera hacia adentro:

  • Workflow: un archivo YAML dentro de .github/workflows/ en tu repo. Define cuándo se ejecuta y qué hace. Podés tener varios.
  • Job: una unidad de trabajo dentro del workflow. Cada job corre en una máquina limpia y separada. Por defecto los jobs corren en paralelo; con needs: los encadenás.
  • Step: un paso dentro de un job. Puede ser un comando de shell (run:) o una action (uses:). Los steps de un job corren en orden, en la misma máquina, y comparten el sistema de archivos.
  • Action: un paso empaquetado y reutilizable, publicado por GitHub o por la comunidad. actions/checkout (clona tu repo) o las setup-* (instalan el JDK, Node, Python, .NET o Go) son actions. Son el "gestor de paquetes" de los pipelines.
  • Runner: la máquina que ejecuta un job. Puede ser de GitHub (hosted) o tuya (self-hosted) — lo vemos en la sección 04.

El "hola mundo" mínimo, con cada parte comentada:

.github/workflows/hola.yml
name: Mi primer workflow        # nombre visible en la pestaña "Actions" del repo

on: push                        # trigger: se ejecuta en cada push a cualquier rama

jobs:
  saludar:                      # id del job (elegís vos el nombre)
    runs-on: ubuntu-latest      # runner: máquina Linux provista por GitHub
    steps:
      - name: Clonar el repositorio
        uses: actions/checkout@v5    # "uses" = ejecutar una action publicada

      - name: Saludar
        run: echo "Hola desde el runner, commit $GITHUB_SHA"   # "run" = comando de shell
Detalle clave

El runner arranca vacío: tu código no está ahí hasta que un step lo clona con actions/checkout. Por eso casi todos los workflows empiezan con ese paso. Y como cada job corre en una máquina distinta, lo que un job genera (tu artefacto) no existe en el siguiente job, salvo que lo pases explícitamente — de eso se ocupan los artifacts (sección 06).

03 · Cuándo se ejecuta

Triggers: on:

El bloque on: define qué eventos de GitHub disparan el workflow. Los cuatro que vas a usar en la práctica:

TriggerSe dispara cuando…Usalo para…
push Alguien sube commits a una rama (o un tag) del repo. El pipeline principal: compilar, testear y desplegar. Filtrá por rama para no correr en todas.
pull_request Se abre o actualiza un PR hacia una rama. Validar el código antes de que entre a la rama: compila + tests, sin desplegar nada.
workflow_dispatch Vos apretás un botón "Run workflow" en la UI de GitHub (o via API). Ejecuciones manuales: re-desplegar, desplegar una versión puntual, correr un rollback.
push a tags Se pushea un tag (ej. v1.4.0). Releases versionados: "cuando taggeo, se construye y publica esa versión exacta".
Los cuatro combinados
on:
  push:
    branches: [main, develop]        # solo pushes a estas ramas
    # también acepta patrones: 'release/**'
    tags: ['v*.*.*']                 # y pushes de tags tipo v1.4.0 (ver nota abajo)
  pull_request:
    branches: [main]                 # PRs que apuntan a main
  workflow_dispatch:                 # botón manual en la UI
    inputs:                          # podés pedir parámetros al que lo ejecuta
      environment:
        description: 'Ambiente destino'
        type: choice
        options: [dev, staging, produccion]

Para tags no hay una clave separada: se filtran dentro del mismo push con tags: ['v*.*.*']. Podés combinar branches: y tags: en el mismo bloque push (matchea si cumple cualquiera de los dos).

Regla práctica

pull_request = verificar sin desplegar. push a ramas = desplegar según la rama. workflow_dispatch = tu botón de emergencia y de control manual. Agregá siempre workflow_dispatch a tu pipeline principal: no cuesta nada y te salva cuando necesitás re-ejecutar un deploy sin hacer un commit vacío.

04 · Dónde se ejecuta

Runners: hosted vs self-hosted

Un runner es la máquina que ejecuta tus jobs. Hay dos mundos, y la elección depende de una pregunta concreta: ¿tus servidores son alcanzables desde internet?

  • GitHub-hosted (runs-on: ubuntu-latest): máquinas virtuales de GitHub, limpias en cada ejecución, con Java, Maven, Git y decenas de herramientas preinstaladas. Cero mantenimiento. Gratis para repos públicos; los privados tienen una cuota mensual de minutos gratis y después se paga por minuto.
  • Self-hosted: una máquina tuya (un servidor en tu red, una VM, incluso el propio servidor de deploy) donde instalás el agente de GitHub. El agente se conecta saliente hacia GitHub (polling por HTTPS) y pregunta "¿hay trabajo para mí?". No hace falta abrir ningún puerto de entrada en tu firewall.
GitHub-hostedSelf-hosted
MantenimientoNingunoVos: OS, actualizaciones, disco, seguridad
Acceso a red privadaNo (está en la nube de GitHub)Sí (está adentro de tu red)
CostoMinutos gratis + pago por uso (privados)Gratis en Actions; pagás tu hardware
EntornoLimpio en cada jobPersistente (¡cuidado con estado residual!)
SeguridadAislado por diseñoNunca lo uses con repos públicos: un PR malicioso ejecutaría código en tu máquina
jobs:
  deploy:
    # hosted:
    runs-on: ubuntu-latest
    # self-hosted (después de registrar el agente en Settings → Actions → Runners):
    # runs-on: [self-hosted, linux, deploy]   # etiquetas que le pusiste al runner
Recomendación para tu caso

Estrategia mixta y muy común: build y tests en runners hosted (aprovechás las máquinas limpias y gratuitas) y solo el job de deploy en un runner self-hosted dentro de tu red, si tus servidores no son públicos. Si tus servidores sí aceptan SSH desde internet (con allowlist de IPs), podés hacer todo con runners hosted y te ahorrás mantener el agente.

05 · Build

Compilar el proyecto y generar el artefacto

Dentro del runner tenés una máquina Linux normal: el build es exactamente el mismo comando que corrés en tu PC. El patrón es siempre igual — clonar el repo, instalar el runtime del lenguaje con su action setup-* (con cache de dependencias), y correr tu comando de build. Lo único que cambia entre stacks es qué se empaqueta: un .war, un bundle comprimido, o un binario.

Job de build — Java con Maven
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Clonar el repo
        uses: actions/checkout@v5

      - name: Instalar JDK 21
        uses: actions/setup-java@v5
        with:
          distribution: temurin      # Eclipse Temurin: la distribución OpenJDK más usada
          java-version: '21'         # poné la versión que use tu proyecto (8, 11, 17, 21…)
          cache: maven               # cachea ~/.m2 → builds mucho más rápidos

      - name: Compilar y empaquetar el WAR
        run: mvn -B -ntp package     # -B: sin colores/prompts (batch), -ntp: sin logs de descarga
        # el .war queda en target/miapp.war (según tu pom.xml)

Con Gradle es igual de directo:

Variante con Gradle
      - uses: actions/setup-java@v5
        with:
          distribution: temurin
          java-version: '21'
          cache: gradle

      - name: Empaquetar el WAR
        run: ./gradlew war --no-daemon    # requiere el plugin 'war' en build.gradle
        # el .war queda en build/libs/
¿Y si mi proyecto necesita una versión exacta de Maven?

Usá Maven Wrapper (./mvnw): commiteás el wrapper al repo y el workflow lo usa igual que en tu máquina (./mvnw -B package). Así todo el equipo y el CI usan exactamente la misma versión de Maven.

Job de build — Node.js
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Clonar el repo
        uses: actions/checkout@v5

      - name: Instalar Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'         # la versión que use tu proyecto (o node-version-file: .nvmrc)
          cache: npm                 # cachea las dependencias descargadas

      - name: Instalar dependencias
        run: npm ci                  # como npm install, pero exacto al package-lock (reproducible)

      - name: Build de producción
        run: npm run build           # si tu app compila (TypeScript, bundler); si no, omitilo

      - name: Empaquetar
        run: tar czf app.tar.gz dist package.json package-lock.json
        # un solo archivo comprimido con lo necesario para correr en el server
        # (las dependencias de producción se instalan en el server con npm ci --omit=dev)
Job de build — Python
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Clonar el repo
        uses: actions/checkout@v5

      - name: Instalar Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'     # la versión que use tu proyecto
          cache: pip                 # cachea los paquetes descargados

      - name: Instalar dependencias
        run: pip install -r requirements.txt

      - name: Empaquetar
        run: tar czf app.tar.gz app/ requirements.txt
        # ajustá "app/" a la estructura de tu proyecto; el venv se crea en el server
        # (Python no compila: el "artefacto" es el código + la lista exacta de dependencias)
Job de build — .NET
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Clonar el repo
        uses: actions/checkout@v5

      - name: Instalar .NET SDK
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'    # la versión de tu proyecto (8 es LTS)

      - name: Publicar (compilar para producción)
        run: dotnet publish -c Release -o publish
        # "publish" deja en ./publish/ todo lo necesario para ejecutar la app

      - name: Empaquetar
        run: tar czf app.tar.gz -C publish .
Job de build — Go
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Clonar el repo
        uses: actions/checkout@v5

      - name: Instalar Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.24'         # la versión de tu go.mod
          # setup-go cachea los módulos automáticamente

      - name: Compilar el binario
        run: CGO_ENABLED=0 go build -o miapp .
        # CGO_ENABLED=0 → binario estático: corre en cualquier Linux sin dependencias
        # el artefacto ES el binario: el caso más simple de todos
Tip

La opción cache: de las actions setup-* guarda las dependencias descargadas entre ejecuciones. Sin cache, cada build vuelve a bajar todo; con cache, un proyecto mediano puede pasar de 6 minutos a 1. Es la optimización con mejor relación costo/beneficio de todo el pipeline.

06 · Pasar archivos entre jobs

Artifacts: mover el artefacto del build al deploy

Acordate: cada job corre en un runner distinto y limpio. Si el job build genera el artefacto y el job deploy intenta leerlo, no lo va a encontrar — son máquinas diferentes. La solución son los artifacts: un almacenamiento temporal que GitHub asocia a la ejecución del workflow. Un job sube el archivo; otro job lo baja. 

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      # ... checkout, setup-java, mvn package ...
      - name: Subir el artefacto
        uses: actions/upload-artifact@v4
        with:
          name: app-package              # nombre con el que lo vas a pedir después
          path: target/*.war         # la ruta de TU artefacto según tu stack (sección 05)
          retention-days: 7          # cuánto lo guarda GitHub (default: 90 días)
          if-no-files-found: error   # si el build no generó el artefacto, fallá acá y no en el deploy

  deploy:
    needs: build                     # espera a que build termine OK
    runs-on: ubuntu-latest
    steps:
      - name: Descargar el artefacto
        uses: actions/download-artifact@v4
        with:
          name: app-package
          path: dist                 # lo deja en ./dist/
Por qué separar build y deploy en jobs distintos

Podrías hacer todo en un job, pero separarlos te da tres cosas: (1) el mismo artefacto exacto se despliega a todos los ambientes — no recompilás para prod, con el riesgo de que salga distinto; (2) podés re-ejecutar solo el deploy si falló la conexión al servidor, sin recompilar; y (3) el deploy puede correr en otro runner (por ejemplo, tu self-hosted dentro de la red privada).

07 · Ambientes

GitHub Environments: dev, staging y producción con reglas

Un Environment es un objeto de GitHub (Settings → Environments) que representa un destino de despliegue. Cuando un job declara environment: produccion, pasan tres cosas:

  1. El job puede leer los secrets y variables propios de ese environment (el host de prod, la clave SSH de prod) — sección 08.
  2. Se aplican las protection rules del environment: la más importante, required reviewers, pausa el job hasta que una persona autorizada apruebe desde la UI. Ahí está tu "botón" de entrega continua.
  3. GitHub registra un historial de deployments: qué commit está en cada ambiente, cuándo y quién lo aprobó.
jobs:
  deploy-prod:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: produccion               # debe existir en Settings → Environments
      url: https://miapp.empresa.com # link "View deployment" en la UI
    steps:
      # este job QUEDA EN PAUSA hasta que un reviewer apruebe,
      # si el environment tiene "Required reviewers" configurado
      - name: Desplegar
        run: echo "desplegando a prod..."

Protection rules disponibles por environment:

  • Required reviewers: hasta 6 personas/equipos; basta que apruebe una.
  • Wait timer: espera fija (ej. 15 min) antes de desplegar — ventana para abortar.
  • Deployment branches: restringe qué ramas pueden desplegar a este environment (ej. solo main puede desplegar a produccion). Esta regla es tu segunda línea de defensa además del if: del workflow.
Ojo con el plan de GitHub

En repos privados, las protection rules de environments (reviewers, wait timer, branch restrictions) requieren plan GitHub Team o Enterprise. En repos públicos están disponibles gratis. Si tu repo es privado con plan Free, igual podés usar environments para separar secrets por ambiente (eso sí funciona); la aprobación manual la simulás con workflow_dispatch para prod.

08 · Configuración sensible

Secrets y variables

Nunca escribas contraseñas, claves SSH o tokens en el YAML: el repo los guardaría en texto plano para siempre (el historial de git no perdona). GitHub ofrece dos almacenes:

  • Secrets (${{ secrets.X }}): cifrados, nunca visibles después de guardarlos, y enmascarados en los logs (si un comando los imprime, salen como ***). Para claves SSH, passwords, tokens.
  • Variables (${{ vars.X }}): configuración no sensible, visible en la UI. Para hostnames, rutas, nombres de app, flags.

Y ambos existen en tres niveles, del más general al más específico:

NivelDónde se defineQuién lo veTípico para…
OrganizaciónSettings de la orgTodos los repos (o una lista)Token del Slack corporativo, credenciales de registry compartido
RepositorioSettings → Secrets and variables → ActionsTodos los workflows del repoCosas comunes a todos los ambientes
EnvironmentSettings → Environments → (nombre)Solo jobs con environment: ese nombreHost, usuario y clave SSH de cada ambiente

Si el mismo nombre existe en varios niveles, gana el más específico: environment > repo > organización. Esto habilita el patrón central de este curso: definís DEPLOY_HOST, DEPLOY_USER y DEPLOY_SSH_KEY en cada environment con valores distintos, y el YAML del deploy es uno solo — el valor correcto se resuelve según a qué environment apunte el job. 

  deploy:
    environment: ${{ needs.mapear.outputs.env_name }}   # dev | staging | produccion
    steps:
      - name: Copiar al servidor
        uses: appleboy/scp-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}      # ← resuelve al host DEL ENVIRONMENT activo
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}    # clave privada SSH (el server tiene la pública)
          source: dist/*
          target: /opt/deploys/incoming/
Nunca

Nunca hagas echo ${{ secrets.X }} "para debuggear", ni lo pases a herramientas que lo puedan loguear. El enmascarado ayuda pero no es infalible (por ejemplo, si el secret se transforma — base64, urlencode — ya no se enmascara).

09 · El deploy en sí

Estrategias para llevar el .war al servidor

El pipeline ya tiene el .war descargado en el runner. Ahora hay que ponerlo en el servidor y activarlo. Las cuatro estrategias habituales:

A · SCP + SSH (la recomendada para empezar)

Copiás el archivo con SCP y ejecutás por SSH un script que lo activa (ubicarlo donde corresponda, reiniciar el servicio, verificar salud). Las actions de la comunidad más validadas para esto son appleboy/scp-action y appleboy/ssh-action (miles de estrellas, mantenimiento activo, v1 estable). 

      - name: Copiar el WAR
        uses: appleboy/scp-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          source: dist/*
          target: /opt/deploys/incoming/
          strip_components: 1            # quita el prefijo "dist/" al copiar

      - name: Activar la nueva versión
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            # versión mínima — el script real, con health check y rollback, está en la sección 12
            sudo systemctl restart miapp   # Java: copiar el .war a webapps y reiniciar tomcat

B · El atajo propio de tu runtime

Además del camino genérico por SSH, cada runtime tiene su mecanismo de deploy "en caliente" o su matiz particular:

Tomcat Manager: Tomcat trae una app de administración (manager) con una API de texto — un curl con el .war lo despliega sin reiniciar el servidor: 

      - name: Deploy via Tomcat Manager
        run: |
          curl --fail -u "${{ secrets.TOMCAT_USER }}:${{ secrets.TOMCAT_PASS }}" \
            -T dist/miapp.war \
            "https://${{ secrets.DEPLOY_HOST }}:8443/manager/text/deploy?path=/miapp&update=true"

Contras: exponés una app de administración (restringila por IP y usá HTTPS), y los redeploys repetidos en caliente pueden dejar fugas de memoria del classloader.

PM2: el process manager más usado para Node en producción. Si tu app corre bajo PM2 en lugar de systemd, el paso de activación por SSH termina en pm2 reload miapp — recarga con cero downtime (levanta el proceso nuevo antes de bajar el viejo). PM2 además reinicia la app si se cae y rota los logs. Es el reemplazo natural de systemd para este stack.

Gunicorn con recarga graceful: si servís la app con Gunicorn (el servidor WSGI estándar), en lugar de systemctl restart podés mandarle kill -HUP al proceso maestro: recarga los workers uno a uno sin cortar conexiones en curso. Para uWSGI existe el equivalente (--touch-reload).

Según el sistema operativo del servidor: en Linux, Kestrel detrás de un reverse proxy (nginx) + systemd es el camino estándar y lo que usa este curso. Si tu servidor es Windows con IIS, la herramienta clásica es Web Deploy (msdeploy) o PowerShell remoto (WinRM) en lugar de SSH/SCP — el resto del pipeline no cambia.

Go es el caso más simple: el binario es autocontenido, no hay runtime ni dependencias que administrar en el servidor. El patrón symlink + systemctl restart es todo lo que necesitás; no existe (ni hace falta) un mecanismo de deploy en caliente estándar. Si querés cero downtime, el camino es un reverse proxy con dos instancias (blue/green).

C · Ansible (cuando hay varios servidores)

Ansible describe el estado deseado del servidor en "playbooks" YAML y lo aplica por SSH. Brilla cuando tenés que desplegar a varios servidores a la vez, con lógica de orquestación (de a uno, health check entre medio, rollback coordinado).

D · Mención: contenedores

El camino moderno es empaquetar la app como imagen Docker (runtime + app adentro) y desplegar la imagen. Te lo dejo señalado como evolución futura: los conceptos de este curso (pipeline, environments, secrets, aprobaciones) se mantienen idénticos.

EstrategiaProsContras
SCP + SSH Simple, transparente, sin nada extra en el server, control total del script El script de activación lo escribís vos (incluido el rollback); requiere acceso SSH desde el runner
Atajo del runtime Usa el mecanismo nativo de tu stack (Tomcat Manager, PM2, gunicorn…): recarga rápida, a veces sin downtime Específico de cada runtime; algunos exponen superficies de administración que hay que proteger
Ansible Multi-servidor, idempotente, orquestación seria, reusable fuera de Actions Otra herramienta que aprender; overkill para un solo servidor
Docker Reproducible, portable, rollback = volver a la imagen anterior Requiere adoptar contenedores en el servidor; cambio de plataforma
Recomendación

Con un servidor por ambiente: SCP + SSH con un script de activación versionado (lo armamos completo en la sección 12 y en el workflow final). Cuando pases de ~3 servidores o necesites orquestación, migrá el paso de activación a Ansible sin tocar el resto del pipeline.

10 · Enrutamiento

Mapear ramas a ambientes

La convención más usada (GitFlow simplificado) asigna cada rama a un destino:

En el workflow, el mapeo se resuelve en un job chiquito que mira la rama y emite un output con el nombre del environment. Los jobs siguientes lo consumen: 

jobs:
  mapear:
    runs-on: ubuntu-latest
    outputs:
      env_name: ${{ steps.map.outputs.env_name }}
    steps:
      - id: map
        run: |
          case "$GITHUB_REF_NAME" in                # nombre de la rama del push
            main)       echo "env_name=produccion" >> "$GITHUB_OUTPUT" ;;
            release/*)  echo "env_name=staging"    >> "$GITHUB_OUTPUT" ;;
            *)          echo "env_name=dev"        >> "$GITHUB_OUTPUT" ;;
          esac

  deploy:
    needs: [build, mapear]
    environment: ${{ needs.mapear.outputs.env_name }}   # ← un solo job para los 3 ambientes
    runs-on: ubuntu-latest
    steps: [...]
Alternativa más simple

Si preferís evitar el job de mapeo, otra opción válida es trunk-based: todo push a main despliega a dev automáticamente, y los deploys a staging y producción se disparan con tags (v1.4.0) o con workflow_dispatch. Menos ramas para mantener; muy usada en equipos chicos.

12 · Cuando sale mal

Rollback: volver atrás rápido

Un deploy puede fallar de dos maneras: el pipeline falla (SCP no conecta, el servicio no arranca) o —peor— el pipeline termina verde pero la app está rota. Para las dos necesitás la misma base: conservar las versiones anteriores en el servidor y tener un camino probado para reactivarlas.

El patrón estándar: directorio de releases + verificación de salud + vuelta atrás automática.

Script de activación — Java/Tomcat (corre por SSH en el servidor)
#!/usr/bin/env bash
set -euo pipefail                     # cualquier error corta el script

RELEASES=/opt/deploys/releases
WEBAPPS=/opt/tomcat/webapps
TS=$(date +%Y%m%d-%H%M%S)

# 1. Archivar la nueva versión con timestamp (queda el historial)
mv /opt/deploys/incoming/miapp.war "$RELEASES/miapp-$TS.war"

# 2. Guardar referencia a la versión actualmente activa (para poder volver)
CURRENT=$(readlink -f "$RELEASES/current.war" || true)

# 3. Activar la nueva
ln -sfn "$RELEASES/miapp-$TS.war" "$RELEASES/current.war"
cp -f "$RELEASES/current.war" "$WEBAPPS/miapp.war"
sudo systemctl restart tomcat

# 4. Health check: esperar hasta 60s a que la app responda
for i in $(seq 1 12); do
  sleep 5
  if curl -sf http://localhost:8080/miapp/health > /dev/null; then
    echo "✓ Deploy OK: miapp-$TS.war"
    # borrar releases viejos, conservando los últimos 5
    ls -t "$RELEASES"/miapp-*.war | tail -n +6 | xargs -r rm -f
    exit 0
  fi
done

# 5. La app no levantó → ROLLBACK automático a la versión anterior
echo "✗ Health check falló. Rollback a: $CURRENT"
if [ -n "$CURRENT" ] && [ -f "$CURRENT" ]; then
  ln -sfn "$CURRENT" "$RELEASES/current.war"
  cp -f "$CURRENT" "$WEBAPPS/miapp.war"
  sudo systemctl restart tomcat
fi
exit 1                                # el step falla → el pipeline queda rojo → notificación

Ensayalo

Un rollback que nunca probaste no existe. Probalo en dev a propósito: desplegá una versión con el health check roto y verificá que el script vuelve solo a la anterior.

13 · Avisos

Notificaciones: enterate sin mirar la pestaña Actions

GitHub ya te manda un email automático cuando un workflow tuyo falla (configurable en tu perfil → Notifications). Para avisos a un canal del equipo, el estándar es Slack con la action oficial slackapi/slack-github-action: 

  notificar:
    needs: [build, deploy]
    if: always()                     # corre SIEMPRE, haya fallado o no lo anterior
    runs-on: ubuntu-latest
    steps:
      - name: Aviso a Slack
        uses: slackapi/slack-github-action@v2
        with:
          webhook: ${{ secrets.SLACK_WEBHOOK_URL }}   # Incoming Webhook del canal
          webhook-type: incoming-webhook
          payload: |
            text: "${{ contains(needs.*.result, 'failure') && '🔴 FALLÓ' || '🟢 OK' }} \
            ${{ github.workflow }} · ${{ github.ref_name }} · \
            ${{ github.event.head_commit.message }} \
            https://github.com/${{ github.repository }}/actions/runs/${{ github.run_id }}"

Las tres piezas que hacen funcionar esto:

  • if: always() — sin esto, el job de notificación se saltea cuando algo falla (que es justo cuando más lo querés).
  • needs.*.result — el resultado de cada job previo (success, failure, cancelled, skipped), para armar el mensaje según cómo terminó.
  • El link al run_id — que el mensaje te lleve con un click al log del error.

Para email explícito (más allá del automático de GitHub) existe dawidd6/action-send-mail; para Teams y Discord hay actions equivalentes basadas en webhooks. El patrón es idéntico.

14 · Seguridad

Buenas prácticas de seguridad

Tu pipeline tiene llaves de tus servidores: tratalo como infraestructura crítica.

  • Mínimo privilegio en el token del workflow. Cada run recibe un GITHUB_TOKEN automático. Por defecto puede tener permisos de escritura sobre el repo; bajalo explícitamente: 
    permissions:
  contents: read        # el pipeline solo necesita LEER el código
# y en Settings → Actions → General: "Workflow permissions" → Read repository contents
  • Mínimo privilegio en el servidor. El usuario SSH de deploy no es root: solo escribe en /opt/deploys (y webapps/ si es Java), y su sudoers permite únicamente reiniciar el servicio de la app. Una clave SSH distinta por ambiente, usada solo para esto.
  • Secret scanning + push protection (Settings → Code security): GitHub escanea el repo buscando credenciales filtradas y bloquea el push si detecta una. Gratis en repos públicos; activalo ya.
  • Pinneá las actions de terceros. @v1 es una etiqueta móvil: si al mantenedor le comprometen la cuenta, tu pipeline ejecuta código ajeno con tus secrets. Para actions que tocan secrets, fijá el SHA del commit: 
          - uses: appleboy/ssh-action@ff2b0a4b6a1f6b6c1a2ee4354d1f6a4d3743d1c0   # v1.2.4
        # el comentario documenta a qué versión corresponde el SHA
    Con Dependabot configurado (sección 16), los SHAs se actualizan solos vía PR.
  • Cuidado con pull_request_target y los self-hosted en repos públicos. Son las dos vías clásicas por las que un desconocido ejecuta código en tu infraestructura. Si tu repo es privado y usás pull_request normal, estás bien.
  • Environments como barrera: los secrets de produccion solo existen para jobs que pasaron por sus protection rules. Aunque alguien pushee un workflow malicioso a una rama, no puede leer los secrets de prod sin la aprobación.
  • OIDC para la nube (si algún día desplegás a AWS/Azure/GCP): en lugar de guardar credenciales cloud como secrets, el workflow se autentica con tokens efímeros federados. Es la práctica de referencia actual; para SSH no aplica, pero conviene saber que existe.
15 · No repetirse

Workflows reutilizables y composite actions

Cuando el deploy a dev, staging y prod es el mismo bloque copiado tres veces, cada corrección hay que hacerla tres veces (y te vas a olvidar una). Dos mecanismos para factorizar:

Workflow reutilizable (workflow_call) — reutiliza jobs enteros

.github/workflows/deploy-reusable.yml — la "función"
name: Deploy (reutilizable)

on:
  workflow_call:                     # ← este workflow se invoca desde otros, no por eventos
    inputs:
      environment:
        required: true
        type: string
    secrets:                         # los secrets se declaran y se pasan explícitamente
      DEPLOY_HOST: {required: true}
      DEPLOY_USER: {required: true}
      DEPLOY_SSH_KEY: {required: true}

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - uses: actions/download-artifact@v4
        with: {name: app-package, path: dist}
      - uses: appleboy/scp-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          source: dist/*
          target: /opt/deploys/incoming/
          strip_components: 1
      # ... paso de activación por SSH ...
El workflow principal lo invoca como un job más
jobs:
  build: [...]

  deploy-dev:
    needs: build
    uses: ./.github/workflows/deploy-reusable.yml   # ← "llamada" al workflow
    with:
      environment: dev
    secrets: inherit          # pasa los secrets del contexto (incluidos los del environment)

Composite action — reutiliza grupos de steps

Vive en .github/actions/mi-action/action.yml dentro del repo y agrupa steps (ej: "checkout + setup del lenguaje + cache" que repetís en todos los jobs). Se usa con uses: ./.github/actions/mi-action. Regla rápida: steps repetidos → composite action; jobs repetidos → workflow reutilizable.

Nota: en nuestro workflow final logramos no repetirnos de una tercera manera — un único job de deploy cuyo environment: es dinámico. Para un solo repo con un flujo lineal alcanza y sobra; los reutilizables ganan cuando tenés varios repos o los deploys difieren entre ambientes.

16 · Lo que nadie te contó

Temas que te faltaban (y te van a morder si los ignorás)

  • Concurrency — el bug más traicionero. Dos pushes seguidos = dos pipelines desplegando al mismo tiempo al mismo servidor. Evitalo declarando un grupo de concurrencia: 
    concurrency:
  group: cicd-${{ github.ref }}     # un pipeline por rama a la vez
  cancel-in-progress: false         # el segundo espera; no cancela un deploy a medias
  • Branch protection + required checks. Settings → Branches → protegé main: exigí PR con revisión y que el check de build/tests esté verde para mergear. Es el candado que hace que "main siempre deployable" sea verdad.
  • Dependabot para las actions. Un archivo y te olvidás de perseguir versiones:
    .github/dependabot.yml
    version: 2
updates:
  - package-ecosystem: github-actions   # PRs automáticos cuando checkout, setup-java, etc. publican versión nueva
    directory: /
    schedule: {interval: weekly}
  - package-ecosystem: maven            # el de tu stack: maven, npm, pip, nuget o gomod
    directory: /
    schedule: {interval: weekly}
  • Timeouts. Por defecto un job puede colgarse hasta 6 horas (y consumir minutos). Poné timeout-minutes: 15 en cada job.
  • Badge de estado en el README: ![CI](https://github.com/usuario/repo/actions/workflows/ci-cd.yml/badge.svg) — el semáforo del proyecto a la vista.
  • Versionar el artefacto. Meté el SHA del commit en el nombre del paquete o en un archivo de versión adentro: cuando algo falle en prod, saber exactamente qué commit está corriendo no tiene precio.
  • Debug de workflows. Para YAML inválido antes de pushear: actionlint (linter de la comunidad, muy adoptado) y la extensión oficial "GitHub Actions" para VS Code (autocompletado y validación del schema). Para runs fallando: re-ejecutá con "Enable debug logging" activado.
  • Los minutos cuestan (en repos privados). El cache de dependencias, los timeouts y evitar builds redundantes (paths-ignore: ['**.md'] en el trigger) mantienen la cuota a raya.

        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            set -euo pipefail
            RELEASES=/opt/deploys/releases
            TS=$(date +%Y%m%d-%H%M%S)

            mv /opt/deploys/incoming/miapp "$RELEASES/miapp-$TS"
            chmod +x "$RELEASES/miapp-$TS"

            CURRENT=$(readlink -f "$RELEASES/current" 2>/dev/null || true)
            ln -sfn "$RELEASES/miapp-$TS" "$RELEASES/current"
            sudo systemctl restart miapp

            for i in $(seq 1 12); do
              sleep 5
              if curl -sf http://localhost:8080/health > /dev/null; then
                echo "✓ Deploy OK: miapp-$TS"
                ls -t "$RELEASES"/miapp-* | tail -n +6 | xargs -r rm -f
                exit 0
              fi
            done

            echo "✗ Health check falló — rollback"
            if [ -n "$CURRENT" ] && [ -f "$CURRENT" ]; then
              ln -sfn "$CURRENT" "$RELEASES/current"
              sudo systemctl restart miapp
            fi
            exit 1

  # ════════════════════ 4 · NOTIFICACIÓN (siempre corre) ════════════════════
  notificar:
    name: Avisar resultado
    needs: [build, mapear, deploy]
    if: always() && github.event_name != 'pull_request'
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: slackapi/slack-github-action@v2
        with:
          webhook: ${{ secrets.SLACK_WEBHOOK_URL }}
          webhook-type: incoming-webhook
          payload: |
            text: "${{ contains(needs.*.result, 'failure') && ':red_circle: FALLÓ' || ':large_green_circle: OK' }} \
            *${{ github.workflow }}* → ${{ needs.mapear.outputs.env_name }} \
            (`${{ github.ref_name }}`) \
            "
Plan de adopción sugerido

No lo actives todo el primer día. Semana 1: solo el job build (CI puro, cero riesgo). Semana 2: deploy automático a dev. Semana 3: staging con release/*. Recién cuando confíes en el proceso, prendé el deploy a producción con la aprobación manual. Cada etapa te da valor por sí sola.

18 · Otra plataforma

¿Y GitLab? Los mismos conceptos, otra sintaxis

Todo lo conceptual de este curso —qué es CI/CD, separar build de deploy, ambientes, secrets por ambiente, quality gates, rollback, aprobación manual antes de producción— aplica idéntico en GitLab. Lo que cambia es la herramienta: GitLab no usa GitHub Actions sino su propio sistema, GitLab CI/CD, con un único archivo .gitlab-ci.yml en la raíz del repo y una sintaxis diferente. Esta tabla traduce el vocabulario:

.gitlab-ci.yml — deploy (continúa el mismo archivo)
.deploy:                             # nombre con punto = job oculto (plantilla)
  stage: deploy
  image: alpine:3.20                 # imagen mínima; le agregamos el cliente SSH
  before_script:
    - apk add --no-cache openssh-client
    - eval $(ssh-agent -s)
    - echo "$DEPLOY_SSH_KEY" | tr -d '\r' | ssh-add -
    - mkdir -p ~/.ssh
    - ssh-keyscan -H "$DEPLOY_HOST" >> ~/.ssh/known_hosts
  script:
    # el artefacto del build ya está acá (los artifacts viajan solos entre stages);
    # ajustá la ruta a tu stack: target/*.war, app.tar.gz o miapp
    - scp target/*.war "$DEPLOY_USER@$DEPLOY_HOST:/opt/deploys/incoming/"
    - ssh "$DEPLOY_USER@$DEPLOY_HOST" "bash -s" < scripts/activar.sh
    # activar.sh = el script de la sección 12 (health check + rollback), versionado en tu repo

deploy-dev:
  extends: .deploy                   # hereda todo de la plantilla
  environment:
    name: dev
  rules:
    - if: $CI_COMMIT_BRANCH == "develop"    # develop → dev, automático

deploy-staging:
  extends: .deploy
  environment:
    name: staging
  rules:
    - if: $CI_COMMIT_BRANCH =~ /^release\//  # release/* → staging, automático

deploy-prod:
  extends: .deploy
  environment:
    name: produccion
    url: https://miapp.empresa.com
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual                   # ← main → produccion, ESPERA APROBACIÓN MANUAL
      allow_failure: false           # sin esto, el pipeline se marca "passed" antes del click

Los equivalentes del resto del curso, en dos líneas cada uno:

  • Secrets por ambiente (sección 08): Settings → CI/CD → Variables; creá DEPLOY_HOST, DEPLOY_USER y DEPLOY_SSH_KEY tres veces, cada una con Environment scope dev, staging y produccion, marcadas como masked. GitLab elige el valor según el environment: del job — el mismo patrón.
  • Concurrency (sección 16): resource_group: produccion en el job de deploy — nunca dos deploys simultáneos al mismo destino.
  • Notificaciones (sección 13): nativas — Settings → Integrations → Slack (o email on failed pipelines). No hace falta un job de notificación.
  • Branch protection (sección 11/16): Settings → Repository → Protected branches, y "Pipelines must succeed" en Merge requests.
  • Validar el YAML (sección 16): GitLab trae editor con lint integrado — CI/CD → Pipeline editor. No necesitás herramienta externa.
  • Secret scanning (sección 14): agregá el template oficial Security/Secret-Detection.gitlab-ci.yml con include:.
Lo importante

El servidor no se entera de qué plataforma usás: el usuario deploy, los directorios /opt/deploys, el script activar.sh con health check y rollback (sección 12) y el servicio systemd son exactamente los mismos. Si mañana migrás de GitHub a GitLab (o al revés), solo reescribís el YAML del pipeline; toda la infraestructura de deploy queda intacta.

19 · Manos a la obra

Guía de ejercicios: de cero a tu propio pipeline

Leer sobre CI/CD no te hace saber CI/CD — hacerlo, sí. Esta guía tiene 20 ejercicios en 5 niveles que cubren todos los temas del curso, en el mismo orden. Están diseñados para que al terminarlos puedas armar un pipeline completo por tu cuenta y, más importante, entender cada línea de lo que armaste.

Reglas del juego (importan más que los ejercicios)

1 · En orden. Cada nivel asume el anterior. No saltees aunque te parezca fácil.
2 · Escribí el YAML a mano en los niveles 1 a 3, sin copiar y pegar. El error de indentación que vas a cometer (y arreglar) enseña más que diez lecturas.
3 · Predecí antes de ejecutar. Antes de cada push, anotá en una línea qué esperás que pase. Si lo que pasó no coincide, ahí está tu aprendizaje del día.
4 · Un repo laboratorio. Creá un repo laboratorio-cicd y hacé todo ahí: al final va a ser tu chuleta personal, con historial de commits incluido.
5 · Marcá tu progreso con los checkboxes — se guardan en tu navegador.

Nivel 1 · El esqueleto — entender qué es un workflow (sin servidor, sin proyecto: solo un repo vacío)

Nivel 1

Practicás: secciones 02 y 03 · Necesitás: solo un repo en GitHub

Creá el repo laboratorio-cicd y, desde la web o tu editor, el archivo .github/workflows/hola.yml con un workflow que en cada push ejecute un job con dos steps: uno que imprima Hola CI y otro que imprima la fecha (date). Escribilo a mano mirando la sección 02.

Lo lograste si: en la pestaña Actions hay un run verde, y al abrirlo encontrás tus dos mensajes dentro del log del job.

Pista

El error más común es la indentación: steps: va adentro del job, y cada step arranca con -. Si GitHub dice "workflow file issue", te marca la línea exacta. El segundo error más común: guardar el archivo fuera de .github/workflows/ — ahí GitHub ni lo mira.

Nivel 1

Practicás: secciones 02 y 04

Antes de correrlo, predecí: ¿qué va a mostrar ls -la si lo ejecutás antes del checkout? Ahora armá un workflow con esta secuencia de steps: pwd y ls -la, después uname -a y la versión de tu runtime (java -version, node -v, python --version…), después actions/checkout@v5, y de nuevo ls -la.

Lo lograste si: podés explicar con tus palabras (1) por qué el primer ls está vacío y el segundo no, y (2) por qué tu runtime ya estaba instalado sin que hicieras nada.

Pista

Respuestas en la sección 02 ("el runner arranca vacío") y en la 04 (los runners hosted traen decenas de herramientas preinstaladas). Si tu versión de runtime no es la que necesitás, para eso existen las actions setup-*.

Nivel 1

Practicás: sección 03

Modificá tu workflow para que se dispare con push a main, con pull_request hacia main, y con workflow_dispatch que reciba un input nombre y lo salude. Después ejercitá los tres caminos: un push directo, un PR desde una rama nueva, y el botón "Run workflow" con tu nombre.

Lo lograste si: en Actions hay tres runs con tres orígenes distintos (fijate la columna que indica el evento), y el run manual muestra tu saludo personalizado.

Pista

El input se lee con ${{ inputs.nombre }}. Ojo: el botón "Run workflow" recién aparece cuando el workflow con workflow_dispatch ya está en la rama por defecto.

Nivel 1

Practicás: lectura de logs — la habilidad que más vas a usar

Meté un error a propósito: un step que ejecute un comando inexistente (run: comando_que_no_existe) entre dos steps válidos. Predecí: ¿corre el step siguiente? Push, y a leer.

Lo lograste si: (1) encontraste en el log la línea exacta del error y el exit code, (2) confirmaste que el step posterior se salteó, (3) te llegó el mail de GitHub avisando del fallo, y (4) lo arreglaste y el run volvió a verde.

Pista

Este mecanismo — un step falla → el job falla → lo que sigue no corre — es exactamente el "quality gate" de la sección 11. Lo acabás de ver con tus ojos; en el nivel 2 lo vas a usar con tests de verdad.

Nivel 2 · Build y artefactos — el pipeline empieza a producir (necesitás un proyecto de tu stack, aunque sea mínimo)

Nivel 2

Practicás: secciones 05 y 11 · Usá el selector de stack de arriba

Subí al repo un proyecto de tu stack con al menos un test (si no tenés uno, generá el esqueleto mínimo con tu herramienta habitual). Armá el job de build de la sección 05: setup-* con cache, tests, empaquetado. Cuando esté verde, rompé un test a propósito (cambiá un assert) y pusheá.

Lo lograste si: con el test roto el pipeline queda rojo y el paso de empaquetado no llegó a ejecutarse — y podés explicar por qué el orden de los steps es el que hace de gate. Después arreglá el test.

Pista

Los steps corren en orden y el primero que falla corta el job. Por eso tests antes que package. Compará con el ejercicio 4: es el mismo mecanismo, ahora con contenido real.

Nivel 2

Practicás: sección 06

Partí tu workflow en dos jobs: build (compila y sube el artefacto con upload-artifact) y verificar (con needs: build, lo descarga y lista su contenido con ls -la). Predecí primero: ¿qué pasaría si verificar no tuviera needs:? Después sacáselo y comprobalo.

Lo lograste si: (1) el job verificar muestra tu artefacto, (2) lo descargaste también a tu PC desde la página del run, y (3) viste qué error da sin needs: (los jobs corren en paralelo y el artefacto todavía no existe).

Nivel 2

Practicás: secciones 05 y 16

Corré tu build dos veces con cache: activado (la segunda debería aprovecharlo) y una vez con el cache comentado. Anotá la duración de cada run (está en la lista de Actions).

Lo lograste si: tenés los tres tiempos anotados y encontraste en el log del segundo run la línea donde la action dice que hubo cache hit.

Nivel 3 · Ambientes y secrets — la configuración que separa dev de prod (sin servidor todavía: lo simulamos)

Nivel 3

Practicás: sección 08 · Solo con valores de juguete, jamás con secrets reales

Creá un secret PALABRA_SECRETA con un valor inventado (ej. banana42). En un step, imprimila — sí, a propósito, para esto es el valor de juguete. Después imprimila transformada: echo "${{ secrets.PALABRA_SECRETA }}" | rev.

Lo lograste si: la primera sale como *** y la segunda sale legible al revés (24ananab) — acabás de demostrar vos mismo por qué la regla es "nunca imprimir ni transformar secrets": el enmascarado protege el valor exacto, no sus derivados.

Nivel 3

Practicás: secciones 07 y 08

Creá los environments dev y produccion en Settings, y en cada uno una variable SALUDO con valores distintos. Armá un workflow con workflow_dispatch + input choice del ambiente, cuyo job use environment: ${{ inputs.ambiente }} e imprima ${{ vars.SALUDO }}.

Lo lograste si: dos ejecuciones manuales del mismo YAML imprimen saludos distintos según el ambiente elegido. Este es el patrón central del curso: la configuración vive en el ambiente, no en el código.

Nivel 3

Practicás: sección 10

Agregá el job mapear de la sección 10 (el del case con GITHUB_OUTPUT) y un job posterior que use el output como environment: e imprima dónde "desplegaría". Creá la rama develop y pusheá a las dos ramas.

Lo lograste si: el push a develop resolvió dev y el push a main resolvió produccion, y entendés el viaje del dato: GITHUB_OUTPUT → outputs del job → needs del siguiente.

Pista

Si el output llega vacío, el 99% de las veces es porque faltó declarar el bloque outputs: a nivel job, o el id: del step no coincide.

Nivel 3

Practicás: sección 07 · Requiere repo público (o plan Team/Enterprise)

Configurá Required reviewers (vos mismo) en el environment produccion y pusheá a main.

Lo lograste si: el run quedó en estado Waiting, te llegó la notificación de "review required", y al aprobar desde la UI el job continuó. Esa pausa es la diferencia entre entrega continua y despliegue continuo (sección 01) — la acabás de operar.

Si tu repo es privado con plan Free

Las protection rules no van a estar disponibles (sección 07). Alternativa didáctica: hacé que el deploy a producción solo sea posible vía workflow_dispatch con un input confirmar que deba valer exactamente SI, validado en un step que falle si no coincide.

Nivel 4 · Deploy real — del pipeline al servidor (acá aparece el servidor; el ejercicio 12 te explica cómo conseguir uno)

Nivel 4

Practicás: secciones 09 y 14 (la parte servidor) · Sin pipeline todavía

Conseguí un Linux para practicar — cualquiera de estas opciones sirve: (a) una VM local con Multipass o VirtualBox, (b) un contenedor Docker corriendo sshd, (c) un VPS chico (hay opciones gratis o de ~US$5). En él: creá el usuario deploy, generá un par de claves SSH en tu PC (ssh-keygen -t ed25519), poné la pública en authorized_keys del usuario, y creá /opt/deploys/incoming y /opt/deploys/releases.

Lo lograste si: desde tu PC, ssh deploy@tu-servidor 'echo ok' responde ok sin pedirte password, y el usuario deploy NO puede hacer sudo libremente.

Nota sobre VM/contenedor local

Una VM local sirve para practicar toda la parte de servidor, pero los runners de GitHub no van a poder alcanzarla desde internet. Para el ejercicio 13 tenés dos salidas: usar un VPS con IP pública, o registrar tu propia máquina como runner self-hosted (sección 04) que sí llega a tu VM local — configurarlo es un excelente ejercicio extra.

Nivel 4

Practicás: secciones 08 y 09

Cargá DEPLOY_HOST, DEPLOY_USER y DEPLOY_SSH_KEY como secrets del environment dev. Extendé tu workflow: después del build, un job de deploy que copie el artefacto con appleboy/scp-action a /opt/deploys/incoming/ y ejecute por SSH un ls -la del directorio.

Lo lograste si: tras cada push a develop, el artefacto aparece en el servidor con fecha nueva — tu primer deploy sin tocar el servidor a mano. Verificalo entrando por SSH vos mismo.

Nivel 4

Practicás: sección 12

Agregale a tu app un endpoint /health que devuelva 200. Implementá el script de activación de la sección 12 (adaptado a tu stack) y hacé un deploy sano para confirmar que todo el ciclo funciona. Ahora la parte valiosa: desplegá a propósito una versión rota — que el health devuelva 500, o que la app no arranque.

Lo lograste si: el pipeline terminó rojo pero tu app siguió respondiendo con la versión anterior. Guardá el log de ese run: es la prueba de que tu rollback funciona antes de necesitarlo en serio. Quien llega hasta acá entendió el 80% del valor de CI/CD.

Pista

Probá el script primero a mano en el servidor (copiá un artefacto a incoming/ y ejecutalo) antes de meterlo al pipeline: depurar bash por SSH desde Actions es lento. Cuando funcione a mano, recién ahí al YAML.

Nivel 4

Practicás: sección 13

Configurá un webhook (Slack, o Discord si no tenés Slack — el patrón es idéntico) y agregá el job notificar con if: always() y mensaje distinto según el resultado. Probá los dos caminos: un run exitoso y uno fallido (ya sabés romper cosas a propósito).

Lo lograste si: recibiste ambos mensajes, el de fallo te llevó con un click al log del error, y podés explicar qué pasaría sin if: always().

Nivel 5 · Nivel producción — lo que diferencia un pipeline de juguete de uno serio

Nivel 5

Practicás: secciones 14 y 16

Cuatro tareas de seguridad e higiene: (1) permissions: contents: read en tu workflow, (2) pin de las actions de terceros por SHA de commit, (3) .github/dependabot.yml para el ecosistema github-actions, (4) branch protection en main exigiendo PR + el check de build en verde.

Lo lograste si: un PR con un test roto queda bloqueado para merge (probalo), y en unos días Dependabot te abre su primer PR de actualización — leelo y mergealo: ese flujo también es parte de CI/CD.

Nivel 5

Practicás: sección 15

Extraé tu lógica de deploy a .github/workflows/deploy-reusable.yml con workflow_call, input environment y secrets: inherit, y llamalo desde el workflow principal.

Lo lograste si: el deploy sigue funcionando igual, tu workflow principal quedó más corto, y un cambio en la lógica de deploy ahora se hace en un solo archivo.

Nivel 5

Practicás: sección 16 (concurrency)

Predecí: ¿qué pasa hoy si hacés dos pushes con 10 segundos de diferencia? Probalo y mirá los dos runs. Después agregá el bloque concurrency con cancel-in-progress: false y repetí.

Lo lograste si: viste la diferencia: sin concurrency los dos corren a la vez (imaginá dos deploys pisándose en el mismo servidor); con concurrency el segundo espera en cola hasta que termina el primero.

Nivel 5

Practicás: sección 18 · Necesitás una cuenta en gitlab.com

Repetí los ejercicios 1, 5 y 13 en GitLab con .gitlab-ci.yml, usando la tabla de equivalencias de la sección 18. Vas a comprobar en carne propia que los conceptos ya los tenés: solo estás traduciendo sintaxis.

Lo lograste si: el pipeline está verde en GitLab y podés nombrar tres diferencias concretas de sintaxis y una cosa que te resultó más simple en cada plataforma.

Nivel 5

Integra: todo el curso

Armá el pipeline de la sección 17 para una app tuya (o el proyecto del laboratorio), sin copiar el YAML entero: construilo por partes, entendiendo cada bloque. Checklist de terminado:

  • Tres environments con sus secrets propios y ramas mapeadas.
  • Tests como gate — nada roto llega a deploy.
  • Un solo artefacto construido una vez y desplegado a todos los ambientes.
  • Aprobación manual (o tu simulación) antes de producción.
  • Rollback automático ensayado (tenés el log del ejercicio 14 que lo prueba).
  • Notificaciones, concurrency, timeout-minutes, permissions mínimos y badge en el README.

El examen final: hacé un cambio trivial de una línea en la app y llevalo a producción usando solamente git push y un click de aprobación — sin tocar el servidor, sin pasos manuales, con aviso automático del resultado.
